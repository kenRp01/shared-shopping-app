# ShareShopi 無料運用の制約

ShareShopi を無料枠中心で運用するための前提、制約、設計ルールをまとめます。
このドキュメントは、機能追加や商用化判断のときに「無料運用を崩さないか」を確認するための基準です。

## 基本方針

- 最初は完全無料に近い構成で開始する。
- 利用者が増えるまでは、重い処理や高頻度通知を入れない。
- 無料枠を超えそうな機能は、実装前に必ず運用コストを確認する。
- 商用化やユーザー増加の前に、有料化または Cloudflare への本格移行を判断する。
- APIキー、シークレット、環境変数の値は GitHub、README、公開ページに書かない。

## 現在の無料運用構成

| 領域 | 現在の構成 | 無料運用上の注意 |
| --- | --- | --- |
| ホスティング | Vercel Hobby | 個人・非商用向け。商用化時は Pro か移行が必要 |
| DB/Auth | Supabase Free | Free プロジェクトは pause される可能性がある |
| メール | Resend Free | 送信数と送信元ドメインに制約がある |
| Cron | Vercel Cron / Cloudflare heartbeat | 高頻度実行は避ける |
| ドメイン | `shareshopi.vercel.app` / `workers.dev` | 独自ドメインは基本的に取得費が必要 |

## 将来の無料優先アーキテクチャ

商用化や Supabase pause 対策を見据え、最終的には以下の構成を候補にします。

| 領域 | 移行候補 | 理由 |
| --- | --- | --- |
| Frontend/API | Cloudflare Workers + OpenNext | Vercel Hobby の商用制約を避けやすい |
| DB | Cloudflare D1 | Cloudflare Workers と相性がよい |
| Auth | Firebase Auth | Googleログイン・メールログインを無料枠で始めやすい |
| Mail | Resend | 問い合わせ、リマインド送信に使う |
| Cron | Cloudflare Cron Triggers | Workers と同じ基盤で日次処理を実行できる |

## 機能制約

無料運用を維持するため、当面は以下の制約を守ります。

| 機能 | 方針 |
| --- | --- |
| リマインド | 1日1回のまとめ通知に限定する |
| 個別時刻通知 | 実装しない。時刻は表示・分類用として扱う |
| メール通知 | 共有メンバー向けの最小限にする |
| 画像アップロード | 実装しない |
| OCR | 実装しない |
| リアルタイム同期 | 実装しない。画面操作時の再取得で対応する |
| SMS認証 | 実装しない |
| 大量ログ保存 | 実装しない |
| 外部分析ツール | 最小限にする |
| 添付ファイル | 実装しない |

## リマインド運用ルール

- 毎日1回だけ実行する。
- 対象は未購入の商品だけにする。
- 同じ日に同じリストへ二重送信しない。
- 送信先は共有メンバーに限定する。
- 公開リンク閲覧者には通知しない。
- `RESEND_API_KEY` または `REMINDER_FROM_EMAIL` が未設定の場合は、送信せず preview 扱いにする。
- 送信数が増えてきたら、通知対象リスト数とメール通数を確認する。

## ゲスト利用の扱い

無料運用では、ゲスト利用を残すことで DB/Auth の利用量を抑えます。

- ログインなしで利用できる。
- 端末内の IndexedDB / localStorage に保存する。
- 共有機能は使えない。
- 端末を変えるとデータは引き継がれない。
- 共有したい場合は Google またはメールログインを求める。

## 共有機能の制約

- 共有は登録済みユーザーへのメール指定、または招待リンク/QRを基本にする。
- 未登録ユーザーへの複雑な招待メール管理は後回しにする。
- 共有メンバーは商品追加・編集・購入済み化ができる。
- リスト削除、公開設定変更、メンバー追加は所有者だけに限定する。
- 公開リンクは閲覧専用にする。

## 無料運用で避けるべき追加

以下はコスト増や設計複雑化につながりやすいため、無料運用中は原則入れません。

- プッシュ通知
- SMS通知
- 画像保存
- レシートOCR
- バーコード読み取り
- リアルタイム共同編集
- 高頻度Cron
- 大量メール配信
- 大容量ログ保存
- 高度な分析ダッシュボード
- 課金機能

## 有料化・移行を検討する基準

以下に当てはまる場合は、無料運用の継続ではなく有料化または移行を検討します。

- Vercel Hobby の商用制約に抵触する。
- Supabase Free の pause が運用リスクになる。
- メール送信数が Resend Free の無料枠に近づく。
- DB容量やクエリ数が無料枠に近づく。
- 利用者から安定稼働を求められる。
- 問い合わせ対応やデータ削除対応が増える。
- 独自ドメインやブランド運用が必要になる。

## 商用化前の最低条件

商用化する場合は、無料運用とは別に以下を確認します。

- Vercel Pro へ上げる、または Cloudflare へ移行する。
- Supabase継続なら Pro 化または pause 対策を行う。
- 独自ドメインを取得する。
- Resend の送信元ドメインを認証する。
- 利用規約、プライバシーポリシー、問い合わせ導線を整える。
- データ削除依頼の運用を決める。
- バックアップ方針を決める。
- E2E テストで主要導線を確認する。

## 判断メモ

短期の限定利用では、現在の構成でも無料に近い運用ができます。
ただし、商用化を考える場合は Vercel Hobby と Supabase Free の制約が大きいため、Cloudflare Workers + D1 + Firebase Auth への移行を優先候補にします。
