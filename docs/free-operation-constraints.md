# ShareShopi 無料運用の制約

ShareShopi を無料枠中心で運用するための前提、制約、設計ルールをまとめます。
このドキュメントは、機能追加や商用化判断のときに「無料運用を崩さないか」を確認するための基準です。

## 基本方針

- 最初は完全無料に近い構成で開始する。
- 利用者が増えるまでは、重い処理や高頻度通知を入れない。
- 無料枠を超えそうな機能は、実装前に必ず運用コストを確認する。
- 商用化やユーザー増加の前に、無料枠の上限と有料化条件を確認する。
- APIキー、シークレット、環境変数の値は GitHub、README、公開ページに書かない。

## 現在の無料運用構成

| 領域 | 現在の構成 | 無料運用上の注意 |
| --- | --- | --- |
| Frontend/API | Cloudflare Workers + OpenNext | 無料枠のリクエスト数とCPU時間を超えないようにする |
| DB | Cloudflare D1 | DB容量、読み書き回数、バックアップ方針を確認する |
| Auth | Firebase Auth | 無料枠で始められるが、本人確認や不正利用対策は別途確認する |
| メール | Resend Free | 送信数と送信元ドメインに制約がある |
| Cron | Cloudflare Cron Triggers | 高頻度実行は避ける |
| ドメイン | `app.shareshopi.workers.dev` | 独自ドメインは基本的に取得費が必要 |

## 無料優先アーキテクチャ

現在は以下の構成を基準にします。

| 領域 | 採用サービス | 理由 |
| --- | --- | --- |
| Frontend/API | Cloudflare Workers + OpenNext | Workers上でWeb/APIをまとめて運用できる |
| DB | Cloudflare D1 | Cloudflare Workers と相性がよい |
| Auth | Firebase Auth | Googleログイン・メールログインを無料枠で始めやすい |
| Mail | Resend | 問い合わせ、リマインド送信に使う |
| Cron | Cloudflare Cron Triggers | Workers と同じ基盤で日次処理を実行できる |

## 機能制約

無料運用を維持するため、当面は以下の制約を守ります。

| 機能 | 方針 |
| --- | --- |
| リマインド | 1日1回のまとめ通知に限定する |
| 個別時刻通知 | 実装しない。時刻は表示・分類用として扱う |
| メール通知 | 共有メンバー向けの最小限にする |
| 画像アップロード | 実装しない |
| OCR | 実装しない |
| リアルタイム同期 | 実装しない。画面操作時の再取得で対応する |
| SMS認証 | 実装しない |
| 大量ログ保存 | 実装しない |
| 外部分析ツール | 最小限にする |
| 添付ファイル | 実装しない |

## リマインド運用ルール

- 毎日1回だけ実行する。
- 対象は未購入の商品だけにする。
- 同じ日に同じリストへ二重送信しない。
- 送信先は共有メンバーに限定する。
- 公開リンク閲覧者には通知しない。
- `RESEND_API_KEY` または `REMINDER_FROM_EMAIL` が未設定の場合は、送信せず preview 扱いにする。
- 送信数が増えてきたら、通知対象リスト数とメール通数を確認する。

## ゲスト利用の扱い

無料運用では、ゲスト利用を残すことで DB/Auth の利用量を抑えます。

- ログインなしで利用できる。
- 端末内の IndexedDB / localStorage に保存する。
- 共有機能は使えない。
- 端末を変えるとデータは引き継がれない。
- 共有したい場合は Google またはメールログインを求める。

## 共有機能の制約

- 共有は登録済みユーザーへのメール指定、または招待リンク/QRを基本にする。
- 未登録ユーザーへの複雑な招待メール管理は後回しにする。
- 共有メンバーは商品追加・編集・購入済み化ができる。
- リスト削除、公開設定変更、メンバー追加は所有者だけに限定する。
- 公開リンクは閲覧専用にする。

## 無料運用で避けるべき追加

以下はコスト増や設計複雑化につながりやすいため、無料運用中は原則入れません。

- プッシュ通知
- SMS通知
- 画像保存
- レシートOCR
- バーコード読み取り
- リアルタイム共同編集
- 高頻度Cron
- 大量メール配信
- 大容量ログ保存
- 高度な分析ダッシュボード
- 課金機能

## 有料化・移行を検討する基準

以下に当てはまる場合は、無料運用の継続ではなく有料化または移行を検討します。

- Cloudflare Workers のリクエスト数やCPU時間が無料枠に近づく。
- D1の容量や読み書き回数が無料枠に近づく。
- メール送信数が Resend Free の無料枠に近づく。
- Firebase Auth の無料枠や本人確認要件に近づく。
- 利用者から安定稼働を求められる。
- 問い合わせ対応やデータ削除対応が増える。
- 独自ドメインやブランド運用が必要になる。

## 週次の無料枠チェック

毎週1回、以下を確認します。数値はサービス側で変わる可能性があるため、最新のDashboard表示を正とします。

| 対象 | 確認先 | 見る項目 |
| --- | --- | --- |
| Workers | `Workers & Pages > app > Metrics` | リクエスト数、CPU時間、エラー率 |
| Cron | `Workers & Pages > app > Settings > Triggers > Cron Events` | 実行成功、失敗、連続失敗 |
| D1 | `Storage & Databases > D1 > shareshopi-prod` | 読み取り回数、書き込み回数、DB容量 |
| Firebase Auth | `Firebase Console > Authentication` | ユーザー数、ログイン失敗、未確認メール |
| Resend | `Resend > Logs` | 送信数、失敗、バウンス、無料枠残量 |

無料枠に近づいた場合は、まず高頻度処理、メール送信数、重いAPI呼び出しを減らします。それでも足りない場合に有料化または構成変更を検討します。

## 商用化前の最低条件

商用化する場合は、無料運用とは別に以下を確認します。

- Cloudflare Workers / D1 / Firebase Auth / Resend の利用量を確認する。
- 無料枠を超える場合の有料化ラインを決める。
- 独自ドメインを取得する。
- Resend の送信元ドメインを認証する。
- 利用規約、プライバシーポリシー、問い合わせ導線を整える。
- データ削除依頼の運用を決める。
- バックアップ方針を決める。
- E2E テストで主要導線を確認する。

## 判断メモ

短期の限定利用では、現在の構成で無料に近い運用ができます。
ただし、商用化を考える場合は各サービスの無料枠、利用規約、送信ドメイン、問い合わせ対応、データ削除対応を必ず確認します。
